Conceptos fundamentales para entender el sistema

¿Qué es una API REST?

API significa Application Programming Interface (Interfaz de Programación de Aplicaciones). Es un conjunto de reglas que permite que dos programas se comuniquen entre sí.

REST (Representational State Transfer) es un estilo de arquitectura que usa los métodos HTTP (GET, POST, PUT, DELETE) para realizar operaciones sobre recursos.

En Happy Pets, la API REST permite que el frontend (la interfaz visual) se comunique con el backend (el servidor) para crear citas, registrar mascotas, generar facturas, etc.

¿Qué es C# y .NET?

C# (pronunciado "C Sharp") es un lenguaje de programación moderno creado por Microsoft. Es orientado a objetos, seguro y muy usado para aplicaciones web, de escritorio y videojuegos.

.NET es la plataforma de desarrollo que ejecuta aplicaciones escritas en C#. Provee bibliotecas, herramientas y un entorno de ejecución común. ASP.NET Core es el framework dentro de .NET para construir aplicaciones web y APIs.

Happy Pets usa .NET 10 y ASP.NET Core, lo que garantiza alto rendimiento y seguridad.

¿Qué es un Endpoint?

Un endpoint es una dirección URL específica de la API que recibe peticiones del cliente. Cada endpoint realiza una función concreta.

Ejemplos en Happy Pets:

  • POST /api/Auth/login → inicia sesión
  • GET /api/Citas → lista citas
  • POST /api/Facturas/cita/5/docx → genera factura

Cada endpoint responde con datos en formato JSON, que el frontend interpreta y muestra al usuario.

¿Qué es JWT?

JWT (JSON Web Token) es un estándar para transmitir información de autenticación entre el cliente y el servidor de forma segura.

Cuando un usuario inicia sesión, el servidor genera un token JWT que el frontend guarda. En cada petición posterior, el frontend envía ese token en los encabezados HTTP para demostrar que está autenticado.

Esto evita tener que enviar usuario y contraseña en cada solicitud y permite proteger los endpoints contra accesos no autorizados.

¿Qué es Entity Framework?

Entity Framework Core (EF Core) es un ORM (Object-Relational Mapper) que permite trabajar con bases de datos usando objetos de C# en lugar de escribir consultas SQL manualmente.

Happy Pets usa Pomelo.EntityFrameworkCore.MySql para conectarse a MySQL. Con EF Core se pueden crear, leer, actualizar y eliminar registros sin escribir SQL directamente, y las migraciones permiten versionar los cambios en la estructura de la base de datos.

¿Qué es SignalR?

SignalR es una biblioteca de ASP.NET Core que permite comunicación en tiempo real entre el servidor y los clientes. A diferencia de una API REST tradicional (donde el cliente debe preguntar si hay novedades), SignalR permite que el servidor envíe datos al cliente instantáneamente cuando ocurre un evento.

En Happy Pets se usa para la mensajería interna entre el personal de la clínica veterinaria.

¿Qué es BCrypt?

BCrypt es un algoritmo de hashing diseñado específicamente para contraseñas. Transforma una contraseña en una cadena irreversible (hash) que se almacena en la base de datos.

Cuando el usuario intenta iniciar sesión, el sistema aplica BCrypt a la contraseña ingresada y compara el resultado con el hash almacenado. Si coinciden, el acceso es concedido. Esto protege las contraseñas incluso si la base de datos es comprometida.

¿Qué es CORS?

CORS (Cross-Origin Resource Sharing) es un mecanismo de seguridad que controla qué dominios externos pueden acceder a los recursos de la API.

Happy Pets configura CORS para permitir únicamente las peticiones provenientes del frontend autorizado, bloqueando accesos desde otros sitios web malintencionados.

Descripción General del Backend
Este backend es la base de la aplicación Happy Pets, diseñada para optimizar el flujo administrativo de una clínica veterinaria: citas, clientes, mascotas, reportes, consultas, facturación y productos. Está construido con .NET 10, ASP.NET Core y C#. Expone una API RESTful documentada con Swagger para facilitar la integración con el frontend. Los repositorios se encuentran en GitHub: github.com/AndresPrograma-desing.

Base de datos MySQL

El sistema utiliza MySQL como motor de base de datos. La cadena de conexión se configura en appsettings.json. Para generar la base de datos se ejecuta:

dotnet ef database update

Esto aplica todas las migraciones pendientes y crea las tablas necesarias.

Migraciones

Las migraciones son archivos que describen cambios en la estructura de la base de datos (crear tablas, añadir columnas, etc.). Se gestionan con Entity Framework Core.

Ejemplo: 20260410014251_InitialCreate.cs contiene la definición inicial de todas las tablas.

Sistema de Facturación — Funcionamiento Detallado

Flujo completo de generación de factura

1. Solicitud desde el frontend: El usuario hace clic en "Generar factura" y el frontend llama al endpoint POST /api/Facturas/cita/{citaId}/docx.

2. Preparación de datos: El servicio InvoiceService recopila los datos de la cita: cliente, mascota, servicios realizados, productos utilizados, precios y totales. Se genera un número de factura único (ej: F-20260410143022-5).

3. Generación del archivo: El servicio GenerateDocxService crea un documento .docx usando OpenXml, con membrete, datos del cliente, lista de items, subtotales y total. El archivo se guarda en wwwroot/facturas/ con un nombre como Factura_Cliente_Mascota_20260410_143022.docx.

4. Respuesta al frontend: El controlador devuelve un JSON con el nombre del archivo y la URL pública: { "file": "Factura_...docx", "url": "https://servidor/facturas/Factura_...docx" }.

5. Persistencia en base de datos: Se guarda (o actualiza) una fila en la tabla facturas con los campos: NumeroFactura, UrlDocx, ConsultaId, ClienteId, MascotaId, Total, FechaEmision, EstadoPago, etc.

6. Descarga por el frontend: El frontend puede mostrar un enlace o botón con la URL recibida, o llamar a GET /api/Facturas/files/download/{fileName} para forzar la descarga.

Limpieza automática (Worker)

Un proceso en segundo plano llamado DeleteFacturaWorker escanea periódicamente la carpeta wwwroot/facturas.

Elimina archivos .docx más antiguos que un umbral de tiempo configurable. Luego actualiza la base de datos: establece UrlDocx = "la factura se ha eliminado" en las filas correspondientes.

Esto libera espacio en disco y notifica al personal mediante mensajería interna.

Consulta de factura existente

El endpoint GET /api/Facturas/cita/{citaId} busca la factura asociada a una cita. Si encuentra el archivo físico en disco, devuelve la URL actualizada. Si el archivo fue eliminado por el worker, devuelve el texto indicando que ya no está disponible.

Estructura de Carpetas del Proyecto

Controllers/

Controladores de la API REST. Cada uno maneja un recurso: AuthController, CitasController, ClientController, PetsController, FacturasController, ReportController, MensajesController.

DTOs/

Data Transfer Objects: clases usadas para recibir y enviar datos en las peticiones HTTP. Separan la capa de presentación de los modelos de base de datos. Ej: CreateCitaDTO, ClientRequestDTO.

Models/

Entidades que representan las tablas de la base de datos: Cita, Cliente, Mascota, Factura, Usuario, Producto, Mensaje.

Services/

Lógica de negocio: AuthService (autenticación), CitasRequestService (gestión de citas), InvoiceService (facturación), GenerateDocxService (creación de documentos).

interface/

Interfaces que definen contratos para los servicios. Facilitan la inyección de dependencias: el sistema puede cambiar la implementación de un servicio sin modificar el código que lo usa.

Configuration/

Clases de configuración como WorkerSetting para parámetros de tareas en segundo plano.

Data/

AppDbContext.cs: contexto de Entity Framework Core que define la conexión a la base de datos y los conjuntos de entidades.

Hubs/

MessageHub.cs: Hub de SignalR para comunicación en tiempo real entre usuarios.

Middleware/

Componentes personalizados que se ejecutan en cada petición HTTP (autenticación, manejo de errores, logging).

Migrations/

Archivos de migración generados por EF Core para versionar los cambios en la base de datos.

Document/

Documentación técnica adicional: FacturationSystem.md, MessagingSystem.md, SecurityAndAuthentication.md.

Worker/

Procesos en segundo plano como DeleteFacturaWorker que ejecuta tareas programadas.

wwwroot/

Archivos estáticos públicos. Aquí se almacenan las facturas generadas en la subcarpeta facturas/.

scripts/

Scripts de administración: call_docx_debug.ps1, create_invoice_test.ps1.

Properties/

launchSettings.json: perfiles de ejecución para desarrollo.

appsettings.json

Archivo principal de configuración: cadena de conexión a MySQL, clave secreta JWT, configuración de CORS, rutas de almacenamiento.

global.json

Especifica la versión del SDK de .NET que requiere el proyecto.

bin/ y obj/

Carpetas generadas automáticamente durante la compilación. Contienen los archivos ejecutables y dependencias compiladas.

Dependencias Principales y su Propósito

BCrypt.Net-Next

Aplica hashing seguro a las contraseñas antes de almacenarlas. Incluye un salt aleatorio para cada contraseña, lo que hace extremadamente difícil revertir el hash incluso con ataques de diccionario.

ClosedXML

Permite crear, leer y modificar archivos Excel (.xlsx) sin necesidad de tener Microsoft Office instalado en el servidor. Se usa para generar reportes exportables.

Pomelo.EntityFrameworkCore.MySql

Proveedor de Entity Framework Core para MySQL. Traduce las operaciones de C# a consultas SQL compatibles con MySQL.

JwtBearer

Middleware de autenticación que valida tokens JWT en cada petición entrante. Si el token es inválido o ha expirado, rechaza la solicitud con un error 401.

Swashbuckle

Genera automáticamente una interfaz Swagger que documenta todos los endpoints de la API, sus parámetros y respuestas. Accesible en /swagger.

QuestPDF

Biblioteca para generar documentos PDF desde cero usando código C#. No requiere dependencias externas ni librerías nativas.

DocumentFormat.OpenXml

Permite manipular archivos de Office (Word, Excel, PowerPoint) a bajo nivel. En Happy Pets se usa para crear los archivos .docx de las facturas.

Seguridad y Autenticación
El sistema implementa JWT para autenticación, BCrypt para hashing de contraseñas, y CORS para restringir el acceso solo al frontend autorizado. También incluye validación de entradas para prevenir inyección SQL y cross-site scripting (XSS).

Flujo de autenticación

1. Usuario envía credenciales al endpoint POST /api/Auth/login.

2. El servidor valida con BCrypt.

3. Si son correctas, genera un token JWT con fecha de expiración.

4. El frontend almacena el token y lo envía en el encabezado Authorization: Bearer {token} en cada petición.

Protección de endpoints

Los controladores usan el atributo [Authorize] para exigir que el usuario esté autenticado. Sin un token JWT válido, el servidor responde con 401 Unauthorized.

Flujo General de la Aplicación
1. Autenticación: Usuario inicia sesión y recibe token JWT.
2. Operaciones CRUD: Los controladores gestionan citas, clientes, mascotas, productos, facturas y reportes.
3. Facturación y Reportes: Se generan facturas en .docx y reportes en PDF/Excel bajo demanda.
4. Mensajería: Los usuarios se comunican en tiempo real mediante SignalR.

Instalación y Ejecución

Pasos para ejecutar

1. Clonar repositorio: git clone <URL>

2. Entrar al directorio: cd backend-happy-pets

3. Restaurar dependencias: dotnet restore

4. Configurar appsettings.json con la cadena de conexión MySQL.

5. Aplicar migraciones: dotnet ef database update

6. Ejecutar: dotnet run

Posibles fallas

  • Tener instalado .NET SDK en la máquina.
  • Verificar que las variables de entorno estén configuradas.
  • Si hay errores de dependencias, limpiar y restaurar: dotnet clean y dotnet restore.
  • Si el antivirus bloquea la ejecución, ejecutar como administrador.
Arquitectura y Diseño
El backend sigue una arquitectura en capas: controladores (presentación), servicios (lógica de negocio) y contexto de datos (acceso a datos). Utiliza inyección de dependencias para desacoplar componentes y facilitar pruebas y mantenimiento. Los patrones Repositorio y Servicio abstraen el acceso a datos y la lógica de negocio respectivamente.
Para más detalles, consulta la documentación Swagger en /swagger y los archivos .md en la carpeta Document/ del repositorio.
