Conceptos fundamentales para entender el sistema

¿Qué es React?

React es una biblioteca de JavaScript para construir interfaces de usuario interactivas. Organiza la aplicación en componentes reutilizables: cada pieza visual (botón, formulario, tabla) es un componente independiente con su propia lógica y diseño.

Happy Pets usa React 19 para crear paneles de administrador, doctor y secretaria de forma modular y eficiente.

¿Qué es Vite?

Vite es una herramienta de desarrollo y empaquetado ultrarrápida. Compila el código React, recarga la página en caliente (Hot Module Replacement) y optimiza los archivos para producción.

Happy Pets usa Vite 8 para levantar el entorno de desarrollo con comandos como npm run dev y construir la versión final con npm run build.

¿Qué es Node.js y Express?

Node.js permite ejecutar JavaScript en el servidor. Express es un framework para Node.js que facilita la creación de APIs y servidores web.

El proyecto incluye un backend mock en la carpeta server basado en Express para probar autenticación y mensajería sin necesidad del backend real.

¿Qué es Polling?

Polling es una técnica donde el frontend consulta periódicamente al backend para obtener datos actualizados. Es como preguntar "¿hay novedades?" cada cierto tiempo.

Happy Pets usa polling en citas, dashboard, facturas y mensajería para mantener la información fresca sin necesidad de que el usuario recargue la página manualmente.

¿Qué es un Token Bearer?

El token Bearer (JWT) es un código de seguridad que el frontend envía en cada petición al backend para demostrar que el usuario está autenticado. Se guarda en sessionStorage/localStorage tras iniciar sesión.

Si el token expira o es inválido, el backend responde con un error 401 y el frontend redirige automáticamente al login.

¿Qué es una API?

API (Application Programming Interface) es el conjunto de reglas que permite que el frontend se comunique con el backend. El frontend envía peticiones HTTP (GET, POST, PUT, DELETE) y recibe respuestas en JSON.

Happy Pets centraliza todas las llamadas a la API en la carpeta services para mantener el código organizado y evitar duplicaciones.

¿Qué son CSS Modules?

CSS Modules es una técnica que permite escribir estilos CSS encapsulados por componente. Cada clase se convierte en única automáticamente, evitando conflictos de nombres entre componentes.

Happy Pets usa CSS Modules junto con estilos globales para mantener una apariencia consistente y modular.

¿Qué es sessionStorage/localStorage?

Son mecanismos del navegador para almacenar datos en el dispositivo del usuario. sessionStorage guarda datos mientras la pestaña esté abierta; localStorage persiste incluso al cerrar el navegador.

Happy Pets replica token y datos de sesión en ambos para garantizar disponibilidad y detectar cuentas bloqueadas.

Descripción General del Sistema
Happy Pets Vet System WEB es el frontend principal del ecosistema de gestión veterinaria. Su objetivo es digitalizar y ordenar el flujo clínico-administrativo diario de la clínica: gestión de usuarios y roles, citas médicas, registro clínico de consultas, productos e inventario, generación y control de facturas, mensajería interna entre usuarios y monitoreo de conectividad. Opera contra una API real configurable por variables de entorno e incluye un backend mock local para pruebas.

Estructura del Proyecto (Carpetas Principales)

public/

Documentación estática en HTML accesible públicamente.

scripts/

Scripts de apoyo para probar endpoints de mensajería y perfil.

server/

Backend mock local basado en Express. Útil para pruebas rápidas de login y chat.

src/

Código principal del frontend.

  • assets: fuentes, íconos, imágenes, audio.
  • components: componentes reutilizables de UI y módulos.
  • config: configuración de interfaz.
  • constants: constantes globales.
  • context: utilidades de estado persistente (user_id).
  • features: funcionalidades agrupadas (mensajería).
  • hooks: hooks custom de datos, polling, salud, notificaciones.
  • pages: páginas por módulo/rol.
  • services: cliente API y funciones de acceso al backend.
  • styles: estilos globales y variables CSS.
  • utils: utilidades de errores API, facturación y helpers.

test/

HTML de pruebas y visualizaciones auxiliares.

Comunicación entre Frontend y Backend
La comunicación está centralizada en la capa de servicios. La base URL es configurable con la variable de entorno VITE_API_URL. Se normalizan rutas automáticamente, se inyecta el token Bearer, se manejan respuestas JSON y no JSON, y se hace logout automático ante error 401. El sistema también monitorea la conectividad mediante un health check periódico al backend y el estado de red del navegador.

Autenticación

Endpoints: login, auth/me, auth/update-password. El token se almacena en sessionStorage y localStorage.

Usuarios

Endpoints: listar, crear, habilitar, deshabilitar, eliminar, obtener secretarias.

Citas

Endpoints: listar solicitudes, crear, actualizar estado, eliminar.

Clientes/Mascotas

Endpoints: crear cliente, crear mascota, crear cliente con mascota, búsqueda de mascotas por usuario.

Consultas

Endpoint: crear consulta (signos vitales, diagnóstico, tratamiento, receta, productos usados).

Productos

Endpoints: listar, crear, actualizar, eliminar, disminuir stock.

Facturas

Endpoints: listar, obtener por cita, actualizar estado de pago.

Mensajería

Endpoints: enviar mensaje, conversaciones por par de usuarios, mensajes por usuario, marcar leído.

Instalación y Ejecución

Requisitos previos

Node.js 18 o superior, npm.

Instalación

1. Instalar dependencias del frontend: npm install

2. Instalar dependencias del backend mock: cd server && npm install && cd ..

Variables de entorno

Crear archivo .env con:

VITE_API_URL=https://tu-backend/api

VITE_HEALTH_URL=https://tu-backend/Health

Ejecución en desarrollo

1. Levantar backend mock: cd server && npm start

2. En otra terminal, levantar frontend: npm run dev

Otros comandos

npm run build - construir para producción

npm run preview - previsualizar build

npm run lint - ejecutar ESLint

npm run documentation - abrir documentación

Detalles Relevantes para Entender y Contribuir

Arquitectura funcional

Diseño por módulos y componentes. Hooks reutilizables para lógica transversal (datos, polling, salud, notificaciones). Formularios complejos con validación declarativa (React Hook Form). Capa API unificada en services/ para minimizar duplicación. Polling parametrizable para mantener datos frescos en citas, dashboard, facturas y mensajería.

Gestión de sesión

Token y datos de login se replican en sessionStorage/localStorage. Se guarda user_id para validaciones de estado de cuenta. El sistema detecta cuentas bloqueadas y fuerza retorno a login.

Compatibilidad backend heterogéneo

El frontend contempla diferencias de naming en payloads/respuestas: español e inglés (rol/role, contenido/content). Distintas estructuras de respuesta (data, result, items, arrays directos). Esto facilita integración con APIs no totalmente homogéneas.

Backend mock local

El backend en server/ es un mock para pruebas rápidas: auth por token fijo, perfil de usuario, mensajería en memoria. No reemplaza una API productiva completa pero es útil para validar login/chat localmente.

Calidad y mantenimiento

Linting con ESLint 9. Separación clara de estilos por módulo. Scripts utilitarios para depurar endpoints. Documentación HTML adicional en public/Documentation.

Guía de Contribución
1. Crear rama por feature o fix. 2. Mantener cambios acotados por módulo. 3. Reutilizar la capa de servicios API existente. 4. Evitar llamadas fetch directas fuera de la capa de servicios. 5. Validar flujo por rol afectado antes de abrir PR. 6. Ejecutar lint y build antes de proponer merge.
Proyecto en evolución activa, con módulos operativos y base sólida para seguir expandiendo: permisos por ruta, pruebas automatizadas, normalización de contratos API y mejoras de observabilidad.
